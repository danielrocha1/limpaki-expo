Roadmap Tecnico de Correcao do MVP

Plano de execucao para liberar o MVP com foco em seguranca, consistencia de fluxo, autorizacao, integridade de dados e maturidade minima de producao.

Prioridade executiva: os PRs 1 a 8 formam a fase de bloqueio de deploy. Os PRs 9 e 10 sao altamente recomendados para consolidacao pos-go-live e escala.

Ordem Exata de Implementacao

  • PR 1: Seguranca de WebSocket
  • PR 2: Fluxo de estado de oferta e servico
  • PR 3: Transacoes e integridade de negocio
  • PR 4: Autorizacao por recurso
  • PR 5: DTOs e validacao de entrada
  • PR 6: Revisao de pagamentos
  • PR 7: Upload seguro
  • PR 8: Banco e migrations
  • PR 9: Realtime escalavel
  • PR 10: Operacao e qualidade

PR 1: Fechar Seguranca de WebSocket

Objetivo: remover o risco mais obvio e preparar a base do realtime.

Arquivos:

  • src/config/realtime.js
  • go/src/config/auth.go
  • go/src/handlers/websocket_handler.go
  • go/src/handlers/chat_handler.go
  • go/main.go

Entradas do PR:

  • remover token da query string
  • aceitar token apenas por canal seguro definido
  • validar Origin no upgrade
  • centralizar extracao e autorizacao do socket
  • limpar logs sensiveis do handshake

Commit sugerido: fix(ws): remove jwt from query string and validate websocket origin

PR 2: Corrigir Modelo de Estado de Oferta e Servico

Objetivo: consertar o fluxo principal do produto.

Arquivos:

  • go/src/handlers/offer_handler.go
  • go/src/handlers/service_handler.go
  • go/src/constants/service_status.go
  • src/services/constants.js

Entradas do PR:

  • servico criado no estado correto apos aceite
  • alinhar accept, start, start-with-pin, complete e cancel
  • remover estados impossiveis
  • revisar exibicao de status e acoes no frontend

Commit sugerido: fix(service-flow): normalize offer and service state transitions

PR 3: Transacoes e Integridade de Negocio

Objetivo: evitar estados quebrados no banco.

Arquivos:

  • go/src/handlers/offer_handler.go
  • go/src/handlers/service_handler.go

Entradas do PR:

  • transacao em aceitar oferta
  • transacao em aceitar contraproposta
  • rollback explicito em fluxo multi-entidade
  • reduzir chance de gravacao parcial

Commit sugerido: fix(db): wrap offer acceptance and service creation in transactions

PR 4: Fechar Autorizacao por Recurso

Objetivo: impedir vazamento e alteracao indevida de dados.

Arquivos:

  • go/src/handlers/user_handler.go
  • go/src/handlers/payment_handler.go
  • go/src/routes/routes.go

Entradas do PR:

  • restringir leitura de usuarios e pagamentos
  • aplicar ownership e role check em rotas sensiveis
  • revisar updates perigosos feitos com BodyParser direto em model

Commit sugerido: fix(authz): enforce ownership and restrict sensitive resource access

PR 5: DTOs e Validacao de Entrada

Objetivo: parar de desserializar request direto em model do banco.

Arquivos:

  • go/src/handlers/user_handler.go
  • go/src/handlers/service_handler.go
  • go/src/handlers/offer_handler.go
  • go/src/handlers/payment_handler.go

Entradas do PR:

  • criar structs de input e output
  • validar campos obrigatorios, enums, limites e ranges
  • sanitizar updates
  • padronizar resposta de erro

Commit sugerido: refactor(api): introduce request dto validation for critical handlers

PR 6: Revisar Pagamentos

Objetivo: deixar o MVP com um fluxo de pagamento minimamente confiavel.

Arquivos:

  • go/src/handlers/payment_handler.go
  • go/src/handlers/checkout_handler.go
  • go/src/models/payment.go

Entradas do PR:

  • definir um fluxo unico de pagamento
  • remover ou isolar fluxo morto/inseguro
  • impedir CRUD livre em pagamento
  • ajustar status e tentativas conforme necessidade do MVP

Commit sugerido: fix(payments): consolidate payment flow and restrict payment operations

PR 7: Upload Seguro

Objetivo: reduzir risco operacional e abuso.

Arquivos:

  • go/src/handlers/upload.go

Entradas do PR:

  • validar MIME type
  • validar tamanho maximo
  • limitar quantidade de arquivos
  • preferir storage seguro a anexo bruto por e-mail

Commit sugerido: fix(upload): validate size and file type for document submission

PR 8: Banco e Migrations

Objetivo: tornar deploy previsivel.

Arquivos:

Entradas do PR:

  • remover AutoMigrate do boot de producao

Commit sugerido: chore(db): replace automigrate with versioned migrations

PR 9: Realtime Escalavel

Objetivo: preparar multiplas instancias.

Arquivos:

  • go/src/realtime/hub.go
  • go/src/realtime/offers.go
  • go/src/chat/websocket/hub.go
  • go/src/chat/container.go

Entradas do PR:

  • introduzir EventBus
  • introduzir PresenceStore
  • implementar Redis para presenca e pub/sub
  • manter hub local apenas como registry de conexoes

Commit sugerido: feat(realtime): add redis-backed presence and cross-instance event delivery

PR 10: Operacao e Qualidade

Objetivo: deixar deploy repetivel.

Arquivos:

  • novos arquivos de infra como Dockerfile e workflow CI
  • go/main.go
  • package.json

Entradas do PR:

  • build backend e frontend
  • healthcheck
  • pipeline minima de validacao
  • padronizacao de envs
  • remocao do self-ping

Commit sugerido: chore(infra): add build pipeline and production health checks

Divisao por Fase

  • Fase de desbloqueio de deploy: PR 1 ao PR 8
  • Fase pos-go-live forte: PR 9 e PR 10

Se o time quiser um MVP mais agressivo, o PR 9 pode ficar para depois, mas isso deve ser uma decisao consciente: producao em instancia unica e sem prometer escala horizontal ainda.
